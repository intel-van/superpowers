# 文档审查系统设计

## 概述

为 superpowers 工作流添加两个新的审查阶段：

1. **规范文档审查** — 在头脑风暴之后，编写计划之前
2. **计划文档审查** — 在编写计划之后，实施之前

两者均遵循实施审查所使用的迭代循环模式。

## 规范文档审查器

**目的：** 验证规范是否完整、一致，并准备好进行实施规划。

**位置：** `skills/brainstorming/spec-document-reviewer-prompt.md`

**检查内容：**

| 类别 | 检查内容 |
|----------|------------------|
| 完整性 | TODO、占位符、"待定"、不完整的章节 |
| 覆盖度 | 缺失的错误处理、边界情况、集成点 |
| 一致性 | 内部矛盾、冲突的需求 |
| 清晰度 | 模糊的需求 |
| YAGNI | 未请求的功能、过度设计 |

**输出格式：**
```
## 规范审查

**状态：** 已批准 | 发现问题

**问题（如有）：**
- [章节 X]：[问题] — [为何重要]

**建议（仅供参考）：**
- [不影响批准的改进建议]
```

**审查循环：** 发现问题 → 头脑风暴代理修复 → 重新审查 → 重复直至批准。

**调度机制：** 使用 Task 工具，`subagent_type: general-purpose`。审查器提示模板提供完整提示。头脑风暴技能的控制器负责调度审查器。

## 计划文档审查器

**目的：** 验证计划是否完整、与规范匹配，并具有合理的任务分解。

**位置：** `skills/writing-plans/plan-document-reviewer-prompt.md`

**检查内容：**

| 类别 | 检查内容 |
|----------|------------------|
| 完整性 | TODO、占位符、不完整的任务 |
| 规范对齐 | 计划覆盖规范需求，无范围蔓延 |
| 任务分解 | 任务原子化，边界清晰 |
| 任务语法 | 任务和步骤的复选框语法 |
| 块大小 | 每块不超过 1000 行 |

**块的定义：** 块是计划文档中任务的逻辑分组，以 `## 块 N: <名称>` 标题分隔。编写计划技能根据逻辑阶段（例如"基础"、"核心功能"、"集成"）创建这些边界。每个块应足够独立，以便单独审查。

**规范对齐验证：** 审查器接收：
1. 计划文档（或当前块）
2. 规范文档的路径以供参考

审查器读取两者并比较需求覆盖范围。

**输出格式：** 与规范审查器相同，但范围限定在当前块。

**审查过程（逐块）：**
1. 编写计划创建块 N
2. 控制器使用块 N 内容和规范路径调度计划文档审查器
3. 审查器读取块和规范，返回裁决
4. 如有问题：编写计划代理修复块 N，返回步骤 2
5. 如批准：继续到块 N+1
6. 重复直至所有块批准

**调度机制：** 与规范审查器相同 — Task 工具，`subagent_type: general-purpose`。

## 更新后的工作流

```
头脑风暴 → 规范 → 规范审查循环 → 编写计划 → 计划 → 计划审查循环 → 实施
```

**规范审查循环：**
1. 规范完成
2. 调度审查器
3. 如有问题：修复 → 返回步骤 2
4. 如批准：继续

**计划审查循环：**
1. 块 N 完成
2. 为块 N 调度审查器
3. 如有问题：修复 → 返回步骤 2
4. 如批准：下一个块或实施

## Markdown 任务语法

任务和步骤使用复选框语法：

```markdown
- [ ] ### 任务 1：名称

- [ ] **步骤 1：** 描述
  - 文件：路径
  - 命令：cmd
```

## 错误处理

**审查循环终止：**
- 无硬性迭代限制 — 循环持续直到审查器批准
- 如果循环超过 5 次迭代，控制器应向人类报告以获取指导
- 人类可选择：继续迭代、在已知问题下批准、或中止

**分歧处理：**
- 审查器为建议性质 — 它们标记问题但不阻止
- 如果代理认为审查器反馈不正确，应在修复中解释原因
- 如果同一问题经过 3 次迭代后分歧仍然存在，向人类报告

**格式错误的审查器输出：**
- 控制器应验证审查器输出包含必填字段（状态、问题（如适用））
- 如果格式错误，重新调度审查器并附上格式说明
- 连续 2 次格式错误响应后，向人类报告
