# 工作树全面改造：检测并推迟

**日期：** 2026-04-06
**状态：** 草稿
**工单：** PRI-974
**取代：** PRI-823（Codex App 兼容性）

## 问题

Superpowers 对工作树管理有自己的约定 — 特定路径（`.worktrees/<分支>`）、特定命令（`git worktree add`）、特定清理（`git worktree remove`）。与此同时，Claude Code、Codex App、Gemini CLI 和 Cursor 都提供各自的原生工作树支持，拥有各自的路径、生命周期管理和清理方式。

这造成了三种失败模式：

1. **重复** — 在 Claude Code 上，技能做了 `EnterWorktree`/`ExitWorktree` 已经做的事
2. **冲突** — 在 Codex App 上，技能试图在已管理的工作树内部创建工作树
3. **幽灵状态** — 技能在 `.worktrees/` 创建的工作树对宿主不可见；宿主的 `.claude/worktrees/` 工作树对技能不可见

对于没有原生支持的宿主（Codex CLI、OpenCode、Copilot 独立版），superpowers 填补了真正的空白。技能不应消失 — 它应当在存在原生支持时让路。

## 目标

1. 当存在原生宿主工作树系统时，优先使用它们
2. 继续为缺少工作树支持的宿主提供支持
3. 修复 `finishing-a-development-branch` 中的三个已知错误（#940, #999, #238）
4. 使工作树创建变为可选而非强制（#991）
5. 将硬编码的 `CLAUDE.md` 引用替换为平台中立语言（#1049）

## 非目标

- 每工作树环境约定（`.worktree-env.sh`、端口偏移）— 阶段 4
- 用于路径强制执行的 PreToolUse 钩子 — 阶段 4
- 多仓库工作树文档 — 阶段 4
- 头脑风暴清单的工作树相关更改 — 阶段 4
- `.superpowers-session.json` 元数据跟踪（有趣的 PR #997 想法，v1 不需要）
- 将钩子符号链接到工作树中（PR #965 想法，单独关注点）

## 设计原则

### 检测状态，而非平台

使用 `GIT_DIR != GIT_COMMON` 来确定"我是否已在工作树中？"，而不是嗅探环境变量来识别宿主。这是一个稳定的 git 原语（自 git 2.5, 2015 年起），在所有宿主上通用，且随着新宿主的出现无需维护。

### 声明式意图，规定式回退

技能描述目标（"确保工作在隔离工作空间中完成"）并在可用时优先使用原生工具。它仅在宿主没有原生工作树支持时，作为回退规定具体的 git 命令。步骤 1a 排在第一位，明确命名原生工具（`EnterWorktree`、`WorktreeCreate`、`/worktree`、`--worktree`）；步骤 1b 排在第二位，提供 git 回退。原始规范保持步骤 1a 抽象（"你知道你自己的工具集"），但 TDD 证明当步骤 1a 过于模糊时，代理会锚定在步骤 1b 的具体命令上。明确的工具命名和同意-授权桥接是使偏好可靠所必需的。

### 基于来源的所有权

谁创建了工作树，谁就拥有它的清理权。如果是宿主创建的，superpowers 不碰它。如果是 superpowers 创建的（通过 git 回退），superpowers 清理它。启发式规则：如果工作树位于 `.worktrees/` 或 `~/.config/superpowers/worktrees/` 下，superpowers 拥有它。其他任何位置（`.claude/worktrees/`、`~/.codex/worktrees/`、`.gemini/worktrees/`）都属于宿主。

## 设计

### 1. `using-git-worktrees` SKILL.md 重写

该技能在创建之前新增三个步骤，并简化创建流程。

#### 步骤 0：检测现有隔离

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
BRANCH=$(git branch --show-current)
```

三种结果：

| 条件 | 含义 | 操作 |
|-----------|---------|--------|
| `GIT_DIR == GIT_COMMON` | 普通仓库检出 | 继续步骤 0.5 |
| `GIT_DIR != GIT_COMMON`，有命名分支 | 已在链接工作树中 | 跳到步骤 3（项目设置）。报告："已在 `<路径>` 上的 `<分支名>` 分支的隔离工作空间中。" |
| `GIT_DIR != GIT_COMMON`，分离 HEAD | 外部管理工作树（例如 Codex App 沙盒） | 跳到步骤 3。报告："已在 `<路径>` 的隔离工作空间中（分离 HEAD，外部管理）。" |

步骤 0 不在意谁创建了工作树或哪个宿主在运行。工作树就是工作树，无论来源如何。

**子模块保护：** `GIT_DIR != GIT_COMMON` 在 git 子模块内部也为真。在得出"已在工作树中"的结论之前，检查我们是否不在子模块中：

```bash
# 如果此命令返回路径，我们在子模块中，而非工作树
git rev-parse --show-superproject-working-tree 2>/dev/null
```

如果在子模块中，视为 `GIT_DIR == GIT_COMMON`（继续步骤 0.5）。

#### 步骤 0.5：同意

当步骤 0 发现没有现有隔离（`GIT_DIR == GIT_COMMON`）时，在创建前征询同意：

> "您希望我设置一个隔离的工作树吗？这将保护您当前分支免受更改影响。（y/n）"

如果同意，继续步骤 1。如果不同意，在原位置工作 — 跳过到步骤 3，不创建工作树。

当步骤 0 检测到现有隔离时，此步骤完全跳过（询问已存在的东西没有意义）。

#### 步骤 1a：原生工具（首选）

> 用户已请求隔离工作空间（步骤 0 同意）。检查您的可用工具 — 您是否有 `EnterWorktree`、`WorktreeCreate`、`/worktree` 命令或 `--worktree` 标志？如果有：用户同意创建工作树就是您使用它的授权。立即使用它并跳到步骤 3。

使用原生工具后，跳到步骤 3（项目设置）。

**设计说明 — TDD 修订：** 原始规范使用了一个故意简短、抽象的步骤 1a（"您知道自己的工具集 — 技能无需命名特定工具"）。TDD 验证推翻了这一点：代理锚定在步骤 1b 的具体 git 命令上，忽略了抽象指导（通过率 2/6）。三项更改修复了这一问题（在 GREEN 和 PRESSURE 测试中达到 50/50 通过率）：

1. **明确的工具命名** — 通过名称列出 `EnterWorktree`、`WorktreeCreate`、`/worktree`、`--worktree`，将决策从解释（"我是否有原生工具？"）转变为事实查找（"`EnterWorktree` 在我的工具列表中吗？"）。在没有这些工具的平台上，代理只需检查，找不到任何东西，然后进入步骤 1b。未观察到误报。
2. **同意桥接** — "用户同意创建工作树就是您使用它的授权" 直接回应了 `EnterWorktree` 的工具级护栏（"仅当用户明确要求时"）。工具描述覆盖技能指令（Claude Code #29950），因此技能必须将用户同意表述为工具所需的授权。
3. **红旗条目** — 在红旗部分中命名特定的反模式（"当您有原生工作树工具时使用 `git worktree add` — 这是 #1 错误"）。

文件拆分（将步骤 1b 放在单独技能中）经测试被证明是不必要的。锚定问题通过步骤 1a 文本的质量解决，而非物理分离 git 命令。对照测试使用完整的 240 行技能（所有 git 命令可见）以 20/20 通过。

#### 步骤 1b：Git 工作树回退

当没有原生工具可用时，手动创建工作树。

**目录选择**（优先级顺序）：
1. 检查是否存在 `.worktrees/` 或 `worktrees/` 目录 — 如果找到，使用它。如果两者都存在，`.worktrees/` 优先。
2. 检查是否存在 `~/.config/superpowers/worktrees/<项目>/` 目录 — 如果找到，使用它（向后兼容遗留的全局路径）。
3. 检查项目的代理指令文件（CLAUDE.md、GEMINI.md、AGENTS.md、.cursorrules 或等效文件）中的工作树目录偏好。
4. 默认为 `.worktrees/`。

无交互式目录选择提示。全局路径（`~/.config/superpowers/worktrees/`）不再作为新用户的选择提供，但该位置的现有工作树会被检测并用于向后兼容。

**安全检查**（仅项目本地目录）：

```bash
git check-ignore -q .worktrees 2>/dev/null
```

如果未被忽略，添加到 `.gitignore` 并在继续前提交。

**创建：**

```bash
git worktree add "$path" -b "$BRANCH_NAME"
cd "$path"
```

**钩子感知：** Git 工作树不继承父仓库的钩子目录。通过 1b 创建工作树后，如果主仓库存在钩子目录，则从其符号链接：

```bash
if [ -d "$MAIN_ROOT/.git/hooks" ]; then
    ln -sf "$MAIN_ROOT/.git/hooks" "$path/.git/hooks"
fi
```

这可以防止在工作移至工作树时，提交前检查、linter 和其他钩子静默停止。（来自 PR #965 的想法。）

**沙盒回退：** 如果 `git worktree add` 因权限错误失败，视为受限环境。跳过创建，在当前目录中工作，继续步骤 3。

**步骤编号说明：** 当前技能将步骤 1-4 列为扁平列表。此重新设计使用 0、0.5、1a、1b、3、4。没有步骤 2 — 它是旧的单一"创建隔离工作空间"步骤，现已拆分为 1a/1b 结构。实现者应干净地重新编号（例如，0 → "步骤 0：检测"，0.5 → 在步骤 0 的流程内，1a/1b → "步骤 1"，3 → "步骤 2"，4 → "步骤 3"）或保留当前编号并附上说明。由实现者选择。

#### 步骤 3-4：项目设置和基线测试（不变）

无论通过哪条路径创建的工作空间（步骤 0 检测到现有的、步骤 1a 原生工具、步骤 1b git 回退、或完全没有工作树），执行汇聚：

- **步骤 3：** 自动检测并运行项目设置（`npm install`、`cargo build`、`pip install`、`go mod download` 等）
- **步骤 4：** 运行测试套件。如果测试失败，报告失败并询问是否继续。

### 2. `finishing-a-development-branch` SKILL.md 重写

收尾技能获得环境检测并修复三个错误。

#### 步骤 1：验证测试（不变）

运行项目的测试套件。如果测试失败，停止。不提供完成选项。

#### 步骤 1.5：检测环境（新增）

重新运行与创建步骤 0 相同的检测：

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
```

三个路径：

| 状态 | 菜单 | 清理 |
|-------|------|---------|
| `GIT_DIR == GIT_COMMON`（普通仓库） | 标准 4 选项 | 无工作树需要清理 |
| `GIT_DIR != GIT_COMMON`，有命名分支 | 标准 4 选项 | 基于来源（见步骤 5） |
| `GIT_DIR != GIT_COMMON`，分离 HEAD | 精简菜单：作为新分支推送并创建 PR、保持现状、丢弃 | 无合并选项（无法从分离 HEAD 合并） |

#### 步骤 2：确定基础分支（不变）

#### 步骤 3：展示选项

**普通仓库和命名分支工作树：**

1. 合并回 `<基础分支>` 本地
2. 推送并创建拉取请求
3. 保持分支现状（我稍后处理）
4. 丢弃此工作

**分离 HEAD：**

1. 作为新分支推送并创建拉取请求
2. 保持现状（我稍后处理）
3. 丢弃此工作

#### 步骤 4：执行选择

**选项 1（本地合并）：**

```bash
# 获取主仓库根目录以确保 CWD 安全（错误 #238 修复）
MAIN_ROOT=$(git -C "$(git rev-parse --git-common-dir)/.." rev-parse --show-toplevel)
cd "$MAIN_ROOT"

# 先合并，验证成功后再删除任何内容
git checkout <基础分支>
git pull
git merge <功能分支>
<运行测试>

# 仅在合并成功后：移除工作树，然后删除分支（错误 #999 修复）
git worktree remove "$WORKTREE_PATH"  # 仅当 superpowers 拥有时
git branch -d <功能分支>
```

顺序至关重要：合并 → 验证 → 移除工作树 → 删除分支。旧技能在移除工作树之前删除分支（这失败是因为工作树仍引用该分支）。先移除工作树的简单修复也不对 — 如果合并随后失败，工作目录消失且更改丢失。

**选项 2（创建 PR）：**

推送分支，创建 PR。不清除工作树 — 用户需要它进行 PR 迭代。（错误 #940 修复：移除矛盾的"然后：清理工作树"措辞。）

**选项 3（保持现状）：** 无操作。

**选项 4（丢弃）：** 需要输入"discard"确认。然后移除工作树（如果 superpowers 拥有它），强制删除分支。

#### 步骤 5：清理（更新）

```
if GIT_DIR == GIT_COMMON:
    # 普通仓库，无工作树需要清理
    完成

if 工作树路径在 .worktrees/ 或 ~/.config/superpowers/worktrees/ 下:
    # Superpowers 创建了它 — 我们拥有清理权
    cd 到主仓库根目录       # 错误 #238 修复
    git worktree remove <路径>

else:
    # 宿主创建了它 — 不干预
    # 如果平台提供工作空间退出工具，使用它
    # 否则，保持工作树原样
```

清理仅针对选项 1 和 4 运行。选项 2 和 3 始终保留工作树。（错误 #940 修复。）

**陈旧工作树修剪：** 在任何 `git worktree remove` 之后，运行 `git worktree prune` 作为自愈步骤。工作树目录可能带外删除（例如，通过宿主清理、手动 `rm` 或 `.claude/` 清理），留下导致混乱错误的陈旧注册。一行代码，防止无声腐化。（来自 PR #1072 的想法。）

## 风险

### 步骤 1a 是承载假设 — 已解决

步骤 1a — 代理优先使用原生工作树工具而非 git 回退 — 是整个设计所依赖的基础。如果代理在具有原生支持的宿主上忽略步骤 1a 而进入步骤 1b，检测并推迟完全失败。

**状态：** 此风险在实施过程中出现了。原始抽象步骤 1a（"您知道自己的工具集"）在 Claude Code 上的通过率为 2/6。TDD 门控按设计工作 — 它在任何技能文件被修改前就捕获了失败，防止了损坏的发布。三次 REFACTOR 迭代识别出根本原因（代理锚定在具体命令上，工具描述护栏覆盖技能指令）并产生了在 GREEN 和 PRESSURE 测试中验证为 50/50 的修复。见上文的步骤 1a 设计说明。

**跨平台验证：**

截至 2026-04-06，Claude Code 是唯一具有代理可调用的会话中工作树工具（`EnterWorktree`）的宿主。其他宿主要么在代理启动前创建工作树（Codex App、Gemini CLI、Cursor），要么没有原生工作树支持（Codex CLI、OpenCode）。步骤 1a 是向前兼容的：当其他宿主添加代理可调用的工作树工具时，代理将根据命名示例匹配它们并使用它们，无需技能更改。

| 宿主 | 当前工作树模型 | 技能机制 | 已验证 |
|---------|----------------------|-----------------|--------|
| Claude Code | 代理可调用 `EnterWorktree` | 步骤 1a | 50/50（GREEN + PRESSURE） |
| Codex CLI | 无原生工具（仅 shell） | 步骤 1b git 回退 | 6/6（`codex exec`） |
| Gemini CLI | 启动时 `--worktree` 标志，无代理工具 | 如果带标志启动则步骤 0，否则步骤 1b | 步骤 0：1/1，步骤 1b：1/1（`gemini -p`） |
| Cursor Agent | 面向用户的 `/worktree`，无代理工具 | 如果用户已激活则步骤 0，否则步骤 1b | 步骤 0：1/1，步骤 1b：1/1（`cursor-agent -p`） |
| Codex App | 平台管理，分离 HEAD，无代理工具 | 步骤 0 检测现有 | 1/1 模拟 |
| OpenCode | 仅检测（`ctx.worktree`），无代理工具 | 步骤 1b git 回退 | 未测试（无 CLI 访问） |

**残余风险：**
1. 如果 Anthropic 更改 `EnterWorktree` 的工具描述使其更严格（例如，"不要基于技能指令使用"），同意桥接将失效。值得提交一个问题，请求工具描述容纳技能驱动的调用。
2. 当其他宿主添加代理可调用的工作树工具时，它们可能使用不在步骤 1a 列表中的名称。随着新工具的出现，应更新列表。通用措辞（"工作树或工作空间隔离工具"）提供了一定的前向覆盖。

### 来源启发式规则

`.worktrees/` 或 `~/.config/superpowers/worktrees/` = 我们的，其他任何位置 = 不干预 的启发式规则对当前所有宿主有效。如果未来宿主采用 `.worktrees/` 作为其约定，我们会遇到误报（superpowers 尝试清理宿主拥有的工作树）。类似地，如果用户在没有 superpowers 的情况下手动运行 `git worktree add .worktrees/experiment`，我们会错误地声称所有权。两者风险都很低 — 每个宿主都使用品牌化路径，且手动创建 `.worktrees/` 不太可能 — 但值得注意。

### 分离 HEAD 收尾

为分离 HEAD 工作树提供的精简菜单（无合并选项）对于 Codex App 的沙盒模型是正确的。如果用户因其他原因处于分离 HEAD 状态，精简菜单仍然合理 — 你确实无法在未先创建分支的情况下从分离 HEAD 合并。
